The Phoenix Project

A Novel About IT, DevOps, and Helping Your Business Win

First published in 2013, this novel follows Bill Palmer as he navigates the chaos of Parts Unlimited's failing IT department. Written in the tradition of The Goal by Eliyahu Goldratt.

Chapter I

Introduction

The Phoenix Project, by Gene Kim, Kevin Behr, and George Spafford, is a novel about DevOps and the pain points IT departments face in real-world operations.

Core message: IT management works like a manufacturing line. People see manufacturing as repetitive and IT as creative — but the underlying principles are the same.

Chapter IV — Problem 1

Technical Debt

💻 DEV
deploy code
bug found!
⚙ OPS
↺ vicious cycle — technical debt accruing
Problem

Dev left behind debts that limited future choices. Ops found bugs, pinged back to Dev, Dev spent huge time fixing — a vicious cycle.

Solution

Set up a CI/CD pipeline to automate deployment and reduce the feedback loop. Like Toyota’s SMED — reducing die changes from 3 days to 1 minute.

7
Chapter IV — Problem 2

Work in Progress (WIP)

WIP: 117
To Do
In Progress
Done
Problem

117 active projects, all marked “urgent”. New tasks keep arriving. WIP only grows — nothing finishes.

Solution

Categorize into 4 types: ■ Business ■ Internal ■ Changes (highest) ■ Unplanned (avoid).
Limit WIP — stop accepting new work, focus on what matters. WIP dropped, project succeeded.

8
Chapter IV — Problem 3

Bottleneck

▼ tasks pouring in ⚠ Brent
(jammed here) slow drip ▼
Problem

Brent holds all core knowledge. Every workflow must pass through him. As tasks pile up, the bottleneck worsens.

Solution

1. Train more people — create more “bottles”.
2. Shield Brent from low-priority work. Route unplanned tasks to non-bottleneck staff.

Chapter IV — Problem 4

Communication

◀
Sales
▲
Finance
Security
▶
📦
product breaking!
Problem

Sales wants speed. Security wants safety. Finance wants profit. Everyone pulls in a different direction — the product breaks.

Solution

Communication is key — everywhere and every time. Align on shared goals before execution, not after.

Manufacturing analogy: In a factory, WIP = orders. Too many unfinished orders + new ones arriving = late deliveries. Stop new intake, clear the backlog, and orders ship on time.

Chapter V — The First Way

Flow

DEV
OPS
★
Development Operations Customer

Ensure work always flows left to right — from Development to Operations to the Customer. Maximize throughput of the entire system, not just one part.

☙

Define work clearly — purpose, requirements, volume, time needed.

☙

Reduce batch sizes — use sprints. Finish each sprint without carrying debt.

☙

Limit WIP — stop starting, start finishing.

☙

Synchronize changes — build systems that handle changes together.

Chapter V — The Second Way

Feedback

Create a synchronized feedback loop. A perfect system is impossible, but complex systems can be made safer:

☙

Detect problems quickly — strengthen information flow in the system.

☙

Gather and resolve together — learn from mistakes collectively, don’t fix in isolation.

☙

Share local knowledge — exploit it across the entire organization.

☙

Leaders train leaders — continuously develop capabilities to solve bottlenecks.

Chapter VI

Conclusion

The Phoenix Project demonstrates that IT is not separate from the business — it is the business. Through Bill Palmer’s journey, we learn that the same principles that revolutionized manufacturing can transform IT operations.

Make Work Visible

You cannot manage what you cannot see. Visualize the flow, identify bottlenecks, limit WIP.

Protect Constraints

System throughput is governed by its bottleneck. Guard it fiercely.

Reduce Batch Sizes

Small, frequent deployments beat large, rare releases. The heart of CI/CD.

Learn from Failure

Gather problems, solve them together, spread knowledge across the organization.

Communicate

Department alignment is not optional. It is the foundation of everything.
